1 Interfaces Java

1.1 Java historique

L’idée de base des interfaces en Java “historique”, c’est de définir un type pour des variables, qui pourront référencer des objets de différentes classes, pourvu que ces classes implémentent l’interface, en proposant une définition pour chacune des méthodes citées dans l’interface. C’est un moyen de réaliser le polymorphisme.

Une interface contient la liste des signatures de méthodes qui devront être définies pour les classes concrètes qui implémentent l’interface.

interface Dessinable {
   void dessiner();           // signature
}

class Rond implements Dessinable {
   Rond(double x, double y, double rayon) {
      ....
   }
   @Override
   void dessiner() {          // code de la méthode
      circle(.....);
   }
}

Dessinable d = new Rond(10,20,30);
d.dessiner();
d = new Carre(....);
d.dessiner();

Par commodité, une déclaration d’interface peut aussi héberger des déclarations de constantes.

https://help.semmle.com/wiki/display/JAVA/Constant+interface+anti-pattern

1.2 Généricité

depuis Java 5, les définitions d’interfaces, classes, méthodes etc.s peuvent être génériques, c’est-à-dire paramétrées par des types. C’est le cas par exemple des collections

ArrayList<Personne> invites = new ArrayList<>();

(Les types ne peuvent pas être des types natifs, comme int, double, …)

Exemple de définition :

class PaireImmuable<T> {
   T x, y;
   PaireImmuable(T x, T y) {
      this.x = x;
      this.y = y;
   }
   T getX() {
      return x;
   }
   ...
}

Les paramètres de type s’appellent souvent T, U, V

1.3 Méthodes statiques

Depuis Java 8, les méthodes statiques peuvent être déclarées dans les interfaces, ce qui évite de créer exprès une classe abstraite.

C’est utilisé par exemple pour des méthodes qui sont des fabriques Exemple :

interface Personnel {

    static Personnel creer(String nom, double salaire) {
        if (salaire < 2000) {
            return new Tacheron(nom, salaire);
        } else {
            return new Chef(nom, salaire);
        }
    }
}

avec les deux classes qui implémentent Personnel.

1.4 Méthode par défaut

Aussi depuis Java 8 : introduit une méthode qui s’exprime à partir de méthodes existantes. Exemple jouet :

interface Compteur {
    int get();
    void set(int v);
    
    default add(int n) {
        set(get() + n);
    }
}

Les classes qui implément l’interface Compteur

  • devront implémenter get et set
  • bénéficieront gratuitement d’une implémentation de add.
  • pourront au besoin redéfinir add.

Exemple, dans l’interface Collection, on peut imaginer une implémentation par défaut de isEmpty qui repose sur l’appel à size();

interface Collection<T> {

    int size();
    default boolean isEmpty() {
        return size() == 0;
    }

}

Mais si, une classe qui implémente Collection, possède par exemple il y a un attribut qui indique la taille, elle redéfinir la méthode pour qu’elle soit plus efficace.

2 Classes anonymes

Les classes anonymes sont un moyen de créer, par new, des objets qui implémentent une interface, en indiquant directement le code des méthodes.

exemple, le constructeur d’un thread prend un paramètre un Runnable, qui a une méthode run();

Runnable r = .....;
Thread t = new Thread(r);
t.start();
  1. Pour créer un tel Runnable, on peut définir une classe
class MonAction implements Runnable {
     @Override
    public void run() {
        try {
            for (int i = 1; i <= 5; i++) {
                System.out.println("coucou");
                Thread.sleep(500);
            }
        } catch (InterruptedException ex) {
            System.out.println("Interrompu");
        }
    }
}

et faire

Runnable r = new MonAction() {
  1. Sous forme de classe anonyme, on écrit directement
Runnable r = new Runnable() {
            @Override
            public void run() {
                try {
                    for (int i = 1; i <= 5; i++) {
                        System.out.println("coucou");
                        Thread.sleep(500);
                    }
                } catch (InterruptedException ex) {
                    System.out.println("Interrompu");
                }
            }
        };

sans “botter en touche” vers un fichier source différent.

Un autre avantage des classes anonymes est qu’elles ont accès aux élements définis dans la classe :

String message = "hello";
Thread t = new Thread(new Runnable() {
    @Override
    public void run() {
        afficher(message, 5, 500);
    }
});

ici la méthode statique afficher, la variable message, …

3 Interfaces fonctionnelles et lambda-expressions.

Une notation nouvelle simplifie encore plus les écritures

Thread t = new Thread( 
    () -> { afficher(message, 5, 500); }   // 1
);

Explications

  1. la notation “flèche” représente une fonction avec ses paramètres (à gauche de la flèche) et son résultat.
  2. elle représente la méthode à implémenter dans l’interface.

En effet

  • le constructeur de Thread prend en paramètre un Runnable,
  • l’interface Runnable n’a qu’une seule méthode (run),
  • il n’y a donc pas d’ambiguité, l’expression avec la flèche décrit un objet qui a cette méthode.

La notation “flèche” est-ce qu’on appelle une expression fonctionnelle (ou lambda-expression pour des raisons historiques.

Les interfaces qui n’ont qu’une méthode (non défaut) sont dites fonctionnelles, parce qu’elles ont principalement vocation à représenter une fonction. On les annote par @FunctionalExpression.
